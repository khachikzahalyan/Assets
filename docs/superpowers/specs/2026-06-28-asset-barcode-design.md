## 1. Goal

Every asset gets a **system-generated, globally-unique numeric barcode** at creation time. When the user clicks
**Создать** on `/assets/new`, the asset is saved (appears in the list), a Code 128 barcode is generated, and a
label is **immediately sent to the printer** so it can be physically attached to the device. The barcode is later
read by the existing `/scan` page to pull up the asset's full details.

This is the **generate + print** half of the barcode story. Reading (the `/scan` page) already exists.

## 2. Decisions (locked)

| Decision | Choice |
|---|---|
| Barcode value | **System-generated unique 9-digit number** (e.g. `100309088`), NOT user-entered, NOT derived from invCode. |
| Data model | **New `barcode` field** on Asset — alongside `invCode` + `serial`. Globally unique. **Immutable** after creation. |
| Required | Always present on new assets (auto-generated, so never missing). Legacy assets: `null` until reprinted/backfilled. |
| Symbology | **Code 128**. |
| Print timing | After **Создать**: save → generate → auto-open the browser print dialog (`window.print()`) with a ready label. |
| Group create | Each asset gets its own unique barcode; print a **sheet of N labels** in one print job. |
| Reprint | **«Печать наклейки»** button on the asset detail page. |
| `/scan` | Resolves by **barcode** first, falls back to `invCode` (so old labels / typed inv codes still work). |
| Scope of uniqueness | Triple uniqueness now: **invCode + serial + barcode** — validated in every create/import path (extends the golden rule). |

## 3. Stack alignment

Firebase/Firestore via the repository pattern; Tailwind + shadcn + design tokens; react-i18next (ru/en/hy); no
Supabase / CSS Modules / Zustand. New dep: **`jsbarcode`** (+ `@types/jsbarcode`) for Code 128 rendering.

## 4. Data model & generation

### 4.1 Field
- `Asset.barcode: string | null` (numeric string). `null` only for legacy assets created before this feature.
- `CreateAssetInput` does **NOT** include `barcode` — it is generated by the write repository, never supplied by the UI.
- `UpdateAssetInput` does **NOT** include `barcode` — immutable after creation.

### 4.2 Generation (in the write repository, atomic with uniqueness checks)
- New pure helper `generateBarcodeCandidate(): string` → a 9-digit numeric string with a non-zero first digit
  (range `100000000`–`999999999`). (Uses `Math.random` — allowed in app code.)
- `createAsset`: after the existing invCode/serial uniqueness checks, loop: generate a candidate, check
  `isBarcodeTaken(candidate)`, retry (cap ~10 attempts) until unique; persist it on the new doc; return it on the
  `Asset`. Throw a clear error if it cannot find a unique value within the cap (practically never — 900M space).
- `createAssetsBatch`: generate N candidates unique against existing assets **and** against each other in the batch
  (mirror the existing in-batch dual-uniqueness logic for invCode/serial). Persist + return.
- New repo methods on `AssetWriteRepository` (+ Firestore + InMemory impls):
  - `isBarcodeTaken(barcode: string, exceptId?: string): Promise<boolean>` — mirror `isInvCodeTaken`
    (`where('barcode','==',barcode) limit 2`).
  - `findByBarcode(barcode: string): Promise<Asset | null>` — mirror `findByInvCode`
    (`where('barcode','==',barcode) limit 1`).
- Firestore `toAsset` mapper: map `barcode: (d.barcode as string | null) ?? null`.
- Import path: any asset-creation path must go through the same generation (centralized in the repo), so imported
  assets also get unique barcodes for free. If the import bypasses `createAssetsBatch`, add generation there too.

## 5. Printing

### 5.1 Barcode rendering
- `jsbarcode` renders Code 128 into an `<svg>`. New component `Barcode128` (`src/components/features/assets/label/Barcode128.tsx`):
  takes `value: string`, renders an `<svg ref>` and draws via JsBarcode in a `useEffect` (`format: 'CODE128'`,
  `displayValue: false`). Pure, no app state.

### 5.2 Label
- New component `AssetLabel` (`src/components/features/assets/label/AssetLabel.tsx`): one printable label showing
  the `Barcode128`, the numeric barcode text, the `invCode`, and `brand model` (truncated). Sized for a ~**50×30 mm**
  label (exact size is a Tailwind/inline style constant that can be tuned during the manual print test).

### 5.3 Print host + trigger
- New `LabelPrintHost` (`src/components/features/assets/label/LabelPrintHost.tsx`): given `assets: Asset[]`, renders
  the labels into a portal at `document.body` inside a `.ams-print-label-root` container, then in a `useLayoutEffect`
  calls `window.print()` once and invokes `onAfterPrint`. Single asset → one label; group → N labels (a sheet).
- Print CSS (small block added to `src/index.css`): `@media print { body > #root { display: none } .ams-print-label-root { display: block } } @media screen { .ams-print-label-root { display: none } }` so only the label(s) print.
  Each label uses `break-inside: avoid` so the sheet paginates cleanly.

### 5.4 Wiring on `AssetCreatePage`
- `handleSubmit` (single): after `createAsset` returns `{ value }`, set `printAssets=[value]` instead of navigating
  immediately; mount `LabelPrintHost`; its `onAfterPrint` then runs `onCreated?.(value)` + `navigate('/assets')`.
- `handleSubmitBatch` (group): after `createAssetsBatch` returns `created`, set `printAssets=created`; `onAfterPrint`
  → `navigate('/assets')`.
- Printing must not block the save: the asset is already persisted; if the user cancels the print dialog, we still
  navigate (the `onAfterPrint`/`onafterprint` path fires on both print and cancel).

### 5.5 Reprint on detail
- `AssetDetailPage`: a **«Печать наклейки»** action (only when `asset.barcode != null`) mounts `LabelPrintHost`
  with `[asset]`. No data change.

## 6. `/scan` update
- `ScanPage` resolver: try `findByBarcode(code)`; if `null`, fall back to `findByInvCode(code)`; else not-found toast.
- Update `ScanPage.test.tsx` to inject a repo exposing both methods and assert the barcode-first resolution.

## 7. Error handling
- Barcode generation can't find a unique value within the retry cap → surface a save error (do not persist a dupe).
- Print dialog cancelled → still navigate; no error.
- Legacy asset with `barcode === null` on the detail page → hide the «Печать наклейки» button (nothing to print)
  OR offer «Сгенерировать и напечатать» (out of scope — note as follow-up backfill).
- `jsbarcode` fails to render (bad value) → label shows the numeric text; never crash the page.

## 8. Testing
- `generateBarcodeCandidate`: returns 9-digit, first digit non-zero (seeded/looped assertion).
- `inMemoryAssetRepository`: `createAsset` assigns a unique non-null barcode; `createAssetsBatch` assigns N distinct
  barcodes; `isBarcodeTaken` + `findByBarcode` found/not-found; triple-uniqueness still blocks dup invCode/serial.
- `Barcode128`: renders an `<svg>` with path children for a value (JsBarcode mocked or real in jsdom).
- `AssetCreatePage`: on single create, `LabelPrintHost` mounts with the created asset and navigation happens after
  print (window.print + onafterprint mocked).
- `ScanPage`: barcode-first resolution + invCode fallback.
- i18n: new keys exist in ru/en/hy.

## 9. i18n (new keys)
- `assets` namespace (or `scan`): `label.print` («Печать наклейки»), `label.reprint`, plus any label captions.
  Add to ru/en/hy.

## 10. Out of scope (deferred)
- Backfilling barcodes for pre-existing assets (a one-off migration / a «Сгенерировать» action on legacy detail).
- Configurable label templates / multiple label sizes / dedicated ZPL thermal-printer output.
- A barcode column in the Assets list table.
- QR variant of the label.

## 11. New dependency
- `jsbarcode` (+ `@types/jsbarcode`).
